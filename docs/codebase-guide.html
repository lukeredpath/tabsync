TabSync Codebase Guide

A walkthrough of the HTML, JavaScript, and Alpine.js patterns that make up the app.

Architecture

TabSync is a fully client-side app — no build step, no server, no bundler. The browser loads ES modules directly. The stack is vanilla JS, with Alpine.js handling reactive UI and localStorage for persistence.

Data Layer
library.js

Pure CRUD. Reads and writes a versioned JSON object to localStorage. No Alpine dependency.

Reactive UI
store.js · editor.js

Alpine store and component. Wraps library.js and drives the sidebar, track list, and modal.

Player
player.js

Vanilla DOM module. Bootstraps the YouTube IFrame API and manages playback state.

These three layers communicate exclusively through CustomEvents dispatched on document (see Event Bus below). No module holds a direct reference to another — they're decoupled.

Module loading order in app.js matters:

  1. Register Alpine plugins (Persist, Focus)
  2. Register store and editor component with Alpine
  3. Initialise theme and help panel (plain DOM, no Alpine)
  4. Alpine.start() — Alpine processes the DOM
  5. initPlayer() — YouTube API bootstrap
Why does order matter?
Plugins must be registered before Alpine.start(), and the store/components must be registered before Alpine.start() too — otherwise Alpine will process the DOM before it knows about $store.lib or editorUI and throw errors.

Event Bus

All cross-module communication uses CustomEvents fired on document. The helper in utils.js wraps this:

export function dispatch(name, detail = null) {
  document.dispatchEvent(new CustomEvent(name, detail ? { detail } : undefined));
}

Events always fire on document — not from a specific DOM element — so any module can listen regardless of its position in the DOM tree.

store.js ── tabsync:editor-open ──▶ editor.js editor.js ── tabsync:library-changed ──▶ store.js (reload) store.js ── tabsync:library-changed ──▶ store.js (reload) store.js ── tabsync:track-selected ──▶ player.js editor.js ── tabsync:track-updated ──▶ player.js (update cache) player.js ── tabsync:playback-started ──▶ store.js (collapse sidebar)

tabsync:editor-open store.js (openEditor) editor.js Opens the add/edit modal. detail is { track } to edit, or null to add.

index.html — DOM Structure & Alpine Wiring

The HTML is an app shell — it defines the layout and wires up Alpine directives. There is no server-rendered content; everything displayed in the track list is generated by Alpine at runtime.

The root scope: <body x-data>

<body x-data>

x-data with no value on the body creates an empty Alpine component scope that covers the entire page. This is necessary so that Alpine directives in the body (like @click and :class) have a component context to operate in. The real data lives in $store.lib (a global store), not in this component — but the component scope is required to access the store.

Sidebar: reactive class binding

<aside id="sidebar" :class="{ collapsed: $store.lib.sidebarCollapsed }">

:class is shorthand for x-bind:class. It takes an object where keys are class names and values are boolean expressions. When $store.lib.sidebarCollapsed is true, the collapsed CSS class is added to the element. CSS handles the animation (transition: transform 0.25s ease).

Folder list: x-for

<template x-for="folder in $store.lib.sortedFolders" :key="folder.id">
  <div class="folder-item"
    x-data="{ renaming: false, draftName: '' }"
    :class="{ active: $store.lib.view === folder.id }"
    @click="if (!renaming) $store.lib.setView(folder.id)">
    ...
  </div>
</template>

x-for must be used on a <template> element — Alpine clones its content for each item. Note that each cloned .folder-item has its own x-data for local state (renaming, draftName). This is component-local state that doesn't need to live in the global store. Each folder item manages its own inline rename mode independently.

Track list: x-for with hover-reveal actions

<template x-for="track in $store.lib.filteredTracks" :key="track.id">
  <div class="track-item"
    :class="{ active: $store.lib.selectedTrackId === track.id }"
    @click="$store.lib.selectTrack(track)">

    <div class="track-actions">
      <button @click.stop="$store.lib.openEditor(track)">✎</button>
    </div>
  </div>
</template>

The edit button uses @click.stop — the .stop modifier calls event.stopPropagation() so the click doesn't bubble up to the parent .track-item and trigger selectTrack at the same time.

The .track-actions div is hidden by default in CSS (display: none) and revealed on hover (.track-item:hover .track-actions { display: flex }). This is a CSS concern, not an Alpine concern — no x-show needed when CSS hover state is sufficient.

Search input: x-model

<input type="search" x-model="$store.lib.searchQuery" />

x-model creates a two-way binding between the input value and a piece of Alpine state. Typing in the input updates $store.lib.searchQuery, which automatically re-triggers the filteredTracks computed getter, which re-renders the x-for track list. This is the core Alpine reactive loop.

Editor modal: x-data, x-show, x-trap

<div id="editor-overlay"
  x-data="editorUI"
  x-show="open"
  style="display:none"
  x-trap="open"
  role="dialog" aria-modal="true">

Several things happening here:

  • x-data="editorUI" — mounts the named component registered via Alpine.data('editorUI', ...) in editor.js. The element becomes a self-contained component with its own state and methods.
  • x-show="open" — toggles display: none/block based on the component's open property.
  • style="display:none" — prevents a flash of the modal before Alpine initialises. Alpine sees this and respects it until it takes over.
  • x-trap="open" — from the @alpinejs/focus plugin. While open is true, Tab key focus is trapped inside this element. On close, focus returns to wherever it was before the modal opened.

js/utils.js — Utilities

js/utils.js

Four small, pure utility functions used throughout the app.

uuid()

export function uuid() {
  if (typeof crypto !== 'undefined' && crypto.randomUUID) {
    return crypto.randomUUID();
  }
  // Fallback for older environments
  return 'xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx'.replace(/[xy]/g, c => { ... });
}

Uses the native Web Crypto API when available. The fallback uses Math.random() which is less cryptographically strong but fine for generating track IDs.

extractVideoId(url)

export function extractVideoId(url) {
  const m = url.match(
    /(?:youtube\.com\/(?:[^/]+\/.+\/|(?:v|e(?:mbed)?)\/|.*[?&]v=)|youtu\.be\/)([^"&?/\s]{11})/
  );
  return m ? m[1] : null;
}

Parses any YouTube URL format (youtube.com/watch?v=, youtu.be/, embed URLs) and returns the 11-character video ID. Used in the editor to validate URLs and in library.js track objects which store the ID directly rather than the full URL.

dispatch(name, detail)

export function dispatch(name, detail = null) {
  document.dispatchEvent(new CustomEvent(name, detail ? { detail } : undefined));
}

Fires a CustomEvent directly on document. The detail field is how data is passed with the event — e.g. a track object with tabsync:track-selected. Using document as the target (rather than a specific element) means any listener registered anywhere in the app will receive it.

fetchOEmbed(videoUrl)

export async function fetchOEmbed(videoUrl) {
  try {
    const endpoint = `https://www.youtube.com/oembed?url=${encodeURIComponent(videoUrl)}&format=json`;
    const res = await fetch(endpoint);
    if (!res.ok) return null;
    const data = await res.json();
    return { title: data.title ?? '', author: data.author_name ?? '' };
  } catch {
    return null;
  }
}

YouTube's oEmbed endpoint is CORS-enabled and requires no API key, making it perfect for a client-side-only app. It accepts a video URL and returns metadata including the video title and channel name. Used by the editor to auto-populate the title and artist fields when a URL is pasted in.

js/library.js — Data Layer

js/library.js

The data layer has no Alpine dependency — it's a pure read/write module over localStorage. This keeps the persistence logic isolated and testable independently.

Storage schema

Everything is stored under a single key (tabsync-library) as a versioned JSON envelope:

{
  "version": 1,
  "tracks": [ { "id": "...", "title": "...", ... } ],
  "folders": [ { "id": "...", "name": "...", ... } ]
}

The version field enables migrations. The migrate(data) function is a stub today but is the right place to add transformation logic when the schema changes.

Load / save pattern

function load() {
  try {
    const raw = localStorage.getItem(STORAGE_KEY);
    if (!raw) return empty();
    const parsed = JSON.parse(raw);
    if (!parsed || typeof parsed !== 'object') return empty();
    return migrate(parsed);
  } catch {
    return empty();  // corrupted storage → start fresh
  }
}

Every mutation follows the same pattern: load() → modify → save(). The data is never kept in memory between calls — each function reads fresh from localStorage. This is intentionally simple; there's no in-memory cache to get out of sync.

Track CRUD

export function createTrack(fields) {
  const data = load();
  const now = new Date().toISOString();
  const track = { ...fields, id: uuid(), createdAt: now, updatedAt: now };
  data.tracks.push(track);
  save(data);
  return track;
}

export function updateTrack(id, fields) {
  const data = load();
  const idx = data.tracks.findIndex(t => t.id === id);
  data.tracks[idx] = { ...data.tracks[idx], ...fields, id, updatedAt: new Date().toISOString() };
  save(data);
  return data.tracks[idx];  // returns the updated track
}

Note that updateTrack returns the full updated track object. The editor uses this return value to dispatch tabsync:track-updated with the fresh data, so the player can update its cached track reference without a round-trip back to localStorage.

Folder deletion cascades to tracks

export function deleteFolder(id) {
  const data = load();
  data.folders = data.folders.filter(f => f.id !== id);
  data.tracks = data.tracks.map(t =>
    t.folderId === id ? { ...t, folderId: null } : t
  );
  save(data);
}

Deleting a folder doesn't delete its tracks — they're reassigned to "All tracks" by setting folderId: null.

js/store.js — Alpine Reactive Store

js/store.js

The store is the reactive UI layer wrapping library.js. It's a global Alpine store, meaning any Alpine component anywhere in the DOM can read and write it via $store.lib.

Registration pattern

export function initStore(Alpine) {
  Alpine.store('lib', {
    tracks: [],
    folders: [],
    // ...
  });
}

initStore receives the Alpine instance as an argument rather than importing it, which keeps the module testable and avoids a circular dependency. It's called from app.js before Alpine.start().

Persisted state with Alpine.$persist

view:             Alpine.$persist('all').as('tabsync-view'),
sortBy:           Alpine.$persist('createdAt-desc').as('tabsync-sort'),
sidebarCollapsed: Alpine.$persist(false).as('tabsync-sidebar-collapsed'),

The @alpinejs/persist plugin makes any property automatically sync to/from localStorage. $persist(defaultValue) sets the initial value if none is stored yet; .as('key') sets the localStorage key name. Any time Alpine updates one of these properties, the new value is written to localStorage. On the next page load, the stored value is read back.

Computed getters

get filteredTracks() {
  let t = this.tracks;

  if (this.view === 'favourites') {
    t = t.filter(x => x.favourite);
  } else if (this.view !== 'all') {
    t = t.filter(x => x.folderId === this.view); // folder view
  }

  const q = this.searchQuery.trim().toLowerCase();
  if (q) t = t.filter(x => x.title.toLowerCase().includes(q) || ...);

  t = sortTracks(t, this.sortBy);

  if (this.difficultyFilter !== null) {
    t = t.filter(x => x.difficulty !== null && x.difficulty <= this.difficultyFilter);
  }

  return t;
},

JavaScript getters on an Alpine store object act as computed properties. Alpine tracks which reactive properties the getter accesses (tracks, view, searchQuery, sortBy, difficultyFilter) and re-runs the getter automatically when any of them change. The result is that the track list in the HTML (x-for="track in $store.lib.filteredTracks") always reflects the current filter/sort state with no manual update calls.

Reload pattern

init() {
  this.reload();
  document.addEventListener('tabsync:library-changed', () => this.reload());
  document.addEventListener('tabsync:playback-started', () => { this.sidebarCollapsed = true; });
},

reload() {
  const lib = getLibrary();
  this.tracks = lib.tracks;
  this.folders = lib.folders;
},

The store doesn't mutate library.js state itself — it calls getLibrary() to get a fresh copy from localStorage and replaces its arrays. Because Alpine stores are reactive, replacing this.tracks with a new array triggers a re-render of anything that reads $store.lib.tracks.

All store mutations (and the editor's submit()) dispatch tabsync:library-changed when done. The store's own listener handles the reload. This means the flow for any mutation is always: mutate localStorage → fire event → store reloads → Alpine re-renders.

js/editor.js — Editor Alpine Component

js/editor.js

The editor is an Alpine component (distinct from a store). While the store is global singleton state, a component is a reusable block of state + behaviour tied to a specific DOM element.

Registration

export function initEditorComponent(Alpine) {
  Alpine.data('editorUI', () => ({
    open: false,
    editingTrack: null,
    title: '',
    // ...
  }));
}

Alpine.data('name', factory) registers a named component factory. The factory is an arrow function returning an object — the object's properties become reactive state, and its functions become methods. In the HTML, x-data="editorUI" instantiates this component on that element.

Component lifecycle: init()

init() {
  document.addEventListener('tabsync:editor-open', e => {
    this.openFor(e.detail?.track ?? null);
  });
  document.addEventListener('keydown', e => {
    if (e.key === 'Escape' && this.open) this.tryClose();
  });
},

Alpine calls init() automatically when it mounts the component. This is where you set up event listeners and anything that needs to happen once on creation. Note that document listeners are registered here rather than inline in the HTML — the modal needs to respond to events from outside its DOM subtree.

Two modes: add vs. edit

openFor(track) {
  this.editingTrack = track ?? null;   // null → Add, object → Edit
  this.title  = track?.title  ?? '';
  this.tabUrl = track ? `https://www.youtube.com/watch?v=${track.tabVideoId}` : '';
  // ... populate all fields
  this.open = true;
  this.openSnapshot = this._snapshot();  // record initial state for dirty-check
},

A single method handles both modes. If track is null, it clears all fields (add mode); if it's a track object, it pre-populates them (edit mode). The heading and submit button label are computed from editingTrack:

get heading()     { return this.editingTrack ? 'Edit Track' : 'Add Track'; },
get submitLabel() { return this.editingTrack ? 'Save Changes' : 'Add Track'; },

Dirty-state tracking

_snapshot() {
  return JSON.stringify({
    title: this.title, artist: this.artist,
    tabUrl: this.tabUrl, tabStart: parseFloat(this.tabStart) || 0,
    // ...
  });
},

get isDirty() {
  return this.openSnapshot !== null && this._snapshot() !== this.openSnapshot;
},

When the modal opens, the current field values are serialised to a JSON string and stored as openSnapshot. The isDirty getter compares the current snapshot to the opening snapshot. If they differ, the user has unsaved changes and tryClose() will prompt for confirmation before discarding.

DOM focus with $refs and $nextTick

this.$nextTick(() => {
  const refs = [this.$refs.titleInput, this.$refs.artistInput, this.$refs.tabUrlInput];
  (refs.find(el => el && !el.value) ?? refs[0])?.focus();
});

$refs gives you direct access to DOM elements that have an x-ref="name" attribute. $nextTick defers the callback until after Alpine has processed its DOM updates — here, it waits until x-show="open" has made the modal visible before trying to focus an input (you can't focus a hidden element). The logic finds the first empty field, or defaults to the title input.

js/player.js — YouTube Player

js/player.js

The player is a vanilla DOM module — no Alpine. It manages the YouTube IFrame API and responds to events from the rest of the app.

YouTube IFrame API bootstrap

function bootstrapYouTubeAPI() {
  window.onYouTubeIframeAPIReady = onYouTubeIframeAPIReady;
  const tag = document.createElement('script');
  tag.src = 'https://www.youtube.com/iframe_api';
  document.head.appendChild(tag);
}

The YouTube IFrame API is loaded by injecting a <script> tag at runtime. YouTube's script calls window.onYouTubeIframeAPIReady when it's ready — so setting that global before appending the tag ensures the callback is registered in time.

Pending track guard

document.addEventListener('tabsync:track-selected', e => {
  if (!apiReady) {
    pendingTrack = e.detail;   // API not ready yet — queue it
    setStatus('Loading API…');
    return;
  }
  loadTrack(e.detail);
});

If a user clicks a track before the YouTube script has loaded (slow network), the track is held in pendingTrack. When onYouTubeIframeAPIReady fires, it checks for a pending track and loads it immediately.

Stale-callback guard: loadId

function loadTrack(track) {
  const myLoadId = ++loadId;  // increment global counter, capture local copy
  // ...
}

function makeOnReady(capturedLoadId) {
  return function onPlayerReady() {
    if (capturedLoadId !== loadId) return;  // a newer track was selected — ignore
    // ... seek and enable controls
  };
}

Each call to loadTrack gets a unique ID. When the YouTube player's async onReady callback fires, it checks whether its captured ID still matches the current loadId. If the user switched tracks while the previous video was loading, the old callback sees a mismatched ID and bails out — preventing stale players from interfering with the new track.

Closure-based event handlers

function makeOnStateChange(capturedLoadId) {
  return function onStateChange(event) {
    if (capturedLoadId !== loadId) return;
    if (event.data === YT.PlayerState.ENDED) {
      pause();
      setStatus('Ended');
    }
  };
}

The make* functions are factory functions that return event handler callbacks. This pattern (returning a function from a function) allows each handler to close over its own capturedLoadId independently.

js/app.js — Entry Point

js/app.js

The entry point wires everything together. The order of operations is critical:

import Alpine  from 'https://cdn.jsdelivr.net/npm/alpinejs@3/dist/module.esm.js';
import Persist from 'https://cdn.jsdelivr.net/npm/@alpinejs/persist@3/dist/module.esm.js';
import Focus   from 'https://cdn.jsdelivr.net/npm/@alpinejs/focus@3/dist/module.esm.js';

// 1. Register plugins — must come before store (which uses $persist)
Alpine.plugin(Persist);
Alpine.plugin(Focus);

// 2. Register store and component — must come before Alpine.start()
initStore(Alpine);
initEditorComponent(Alpine);

// 3. Non-Alpine initialisation
initTheme();
initHelp();

// 4. Start Alpine — processes the DOM, mounts components, runs init()
window.Alpine = Alpine;
Alpine.start();

// 5. Bootstrap YouTube API (after Alpine so DOM is ready)
initPlayer();

Why is theme management outside Alpine?

The theme needs to be applied before the page is visible to avoid a flash of the wrong theme. Alpine initialises after the DOM is parsed, which would be too late. The theme code runs synchronously from the module's top level, reads localStorage directly, and sets data-theme on <html> immediately.

Alpine.js — Directives Used in This App

A quick reference for the Alpine directives you'll see in index.html:

DirectiveWhat it doesExample
x-data Creates a component scope. Can be an inline object x-data="{ open: false }", a named component x-data="editorUI", or empty x-data for a store-only scope. <body x-data>
x-model Two-way binding between an input and a reactive property. Reading the input updates the property; changing the property updates the input. x-model="$store.lib.searchQuery"
x-show Toggles display: none based on an expression. The element stays in the DOM (vs. x-if which removes it). x-show="open"
x-if Conditionally adds or removes an element from the DOM. Used inside <template>. Use x-show for frequent toggling; x-if for infrequent (e.g. rename input that only exists while renaming). <template x-if="renaming">
x-for Repeats a <template>'s content for each item in an array. Always provide :key with a unique identifier for efficient DOM diffing. x-for="track in $store.lib.filteredTracks" :key="track.id"
x-text Sets the text content of an element to an expression's value. x-text="track.title"
:attr / x-bind:attr Dynamically sets an HTML attribute. :class is especially useful — accepts an object of { className: booleanExpr }. :class="{ active: $store.lib.view === 'all' }"
@event / x-on:event Attaches an event listener. Modifiers: .stop (stopPropagation), .prevent (preventDefault), .once, key modifiers like @keydown.enter. @click.stop="$store.lib.removeTrack(track.id, track.title)"
x-ref Gives a DOM element a name accessible via this.$refs.name inside the component. x-ref="titleInput" → this.$refs.titleInput.focus()
x-trap From @alpinejs/focus. Traps keyboard focus inside the element while the expression is truthy. Restores focus on close. x-trap="open"

Alpine.js — Magic Properties Used in This App

Magic properties are available inside any Alpine expression (inline HTML or component methods):

MagicWhat it does
$store.lib Access the global store registered as Alpine.store('lib', ...). Readable and writable from any Alpine context anywhere in the DOM.
$refs.name Direct reference to a DOM element with x-ref="name" on the same component. Only available inside component methods, not inline expressions.
$nextTick(fn) Defers fn until after Alpine has finished processing its current batch of DOM updates. Essential when you need to interact with DOM elements that Alpine is about to show or create.
$persist(default) From @alpinejs/persist. Wraps a value so it automatically syncs to/from localStorage. Use .as('key') to name the storage key. Used in the store for view, sortBy, sidebarCollapsed.
$el The current DOM element. Used in the editor backdrop click handler: if ($event.target === $el) — checks whether the click was on the backdrop itself (not a child element).
$event The native DOM event object, available inside @event handlers. Used to read $event.target or $event.target.files.
$dispatch vs dispatch()
Alpine provides a $dispatch(name, detail) magic that fires a CustomEvent from the current element and bubbles it up the DOM. The app uses dispatch() from utils.js instead, which fires directly on document. This matters because Alpine's $dispatch relies on DOM bubbling — if an element were ever in a shadow DOM or portal, it could silently fail to reach document-level listeners.
